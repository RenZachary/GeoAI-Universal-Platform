# SSE Streaming Architecture - Frontend/Backend Alignment

## Date
May 5, 2026

## Executive Summary

Fixed critical mismatch between backend SSE streaming architecture and frontend event handling expectations. The backend was sending workflow-centric events while the frontend expected conversation-centric events, resulting in **zero UI updates** despite successful backend execution.

---

## 🔍 Problem Analysis

### Symptom
- Backend successfully executes chat workflow
- SSE stream contains 20+ events (visible in browser DevTools)
- Frontend displays **nothing** - no messages, no errors, no content

### Root Cause: Architectural Paradigm Mismatch

The system had **two incompatible streaming paradigms**:

#### Backend Architecture (LangGraph-Centric)
```typescript
// What backend sends:
{ type: 'step_start', step: 'goalSplitter' }      // Workflow progress
{ type: 'step_complete' }                           // Step completion
{ type: 'error', message: '...' }                   // Errors
{ type: 'complete', services: [] }                  // Final result
```

**Focus:** Internal workflow execution details for debugging

#### Frontend Architecture (Conversation-Centric)
```typescript
// What frontend expects:
{ type: 'message_start', data: { content: '...' } }  // User message
{ type: 'token', data: { token: '...' } }            // Streaming text
{ type: 'message_complete', data: { summary: '...' } } // Done
{ type: 'error', data: { error: '...' } }            // Errors
```

**Focus:** User-facing conversational experience

## 🏗️ Architectural Issues Identified

### Issue #1: Event Type Vocabulary Mismatch ❌

**Problem:** Backend and frontend use completely different event type names with no translation layer.

**Impact:** Frontend's `handleSSEEvent()` switch statement never matches any backend events → silent failure.

**Code Evidence:**
```typescript
// Frontend: web/src/stores/chat.ts line 73-109
switch (type) {
  case 'message_start':    // ❌ Never received from backend
  case 'token':            // ❌ Never received from backend  
  case 'message_complete': // ❌ Never received from backend
  case 'error':            // ✅ Received but wrong structure
}

// Backend: server/src/llm-interaction/handlers/GeoAIStreamingHandler.ts
this.writeSSE({ type: 'step_start', ... })     // ❌ Frontend doesn't handle
this.writeSSE({ type: 'step_complete', ... })   // ❌ Frontend doesn't handle
this.writeSSE({ type: 'complete', ... })        // ❌ Frontend doesn't handle
```

### Issue #2: Missing Content Streaming ❌

**Problem:** Summary is generated by `SummaryGenerator` node but never sent to frontend.

**Root Cause:** 
- Summary stored in LangGraph state (`state.summary`)
- Controller captures it but doesn't stream it
- Only sends final `complete` event with empty services array

**Code Evidence:**
```typescript
// GeoAIGraph.ts line 315-318

  currentStep: 'summary',

// ChatController.ts line 93-103 (BEFORE FIX)

}

res.write(`data: ${JSON.stringify({
  type: 'complete',
  services: finalServices,  // ❌ No summary field
})}\n\n`);
```

### Issue #3: Duplicate Error Events ❌

**Problem:** Every error appears twice in the stream with identical timestamps.

**Root Cause:** LangGraph calls `handleChainError()` callback twice:
1. From the failing node's execution context
2. From the workflow-level error propagation

**Evidence from Stream:**
```javascript

data: {"type":"error","message":"(f-string)...", "timestamp":1777922273012} // Same ms!

data: {"type":"error","message":"401 Incorrect API key...", "timestamp":1777922273185}
data: {"type":"error","message":"401 Incorrect API key...", "timestamp":1777922273185} // Same ms!
```

### Issue #4: No User Message Display ❌

**Problem:** User's input message never appears in chat UI.

**Root Cause:** Frontend expects `message_start` event to add user message to conversation history, but backend never sends it.

**Code Evidence:**

## ✅ Architectural Fixes Applied

### Fix #1: Bridge Event Type Gap

**File:** [ChatController.ts](file://e:/codes/GeoAI-UP/server/src/api/controllers/ChatController.ts#L43-L59)

**Strategy:** Send frontend-compatible events at key workflow milestones while preserving workflow events for debugging.

```typescript
// Send message_start when conversation begins
res.write(`data: ${JSON.stringify({
  type: 'message_start',
  data: {
    conversationId: convId,
    content: message
  },
  timestamp: Date.now()
})}\n\n`);

// Keep GeoAIStreamingHandler for workflow progress (debugging)
const streamingHandler = new GeoAIStreamingHandler(res);
```

**Result:** Frontend now receives expected event types and can display user messages.

### Fix #2: Stream Summary Content

**File:** [ChatController.ts](file://e:/codes/GeoAI-UP/server/src/api/controllers/ChatController.ts#L111-L145)

**Strategy:** Capture summary from workflow state and stream it as tokens for progressive display.

```typescript
// Capture summary from workflow state
let finalSummary: string = '';

for await (const chunk of stream) {
  // Capture summary from summaryGenerator node
  if (chunk.summaryGenerator && chunk.summaryGenerator.summary) {
    finalSummary = chunk.summaryGenerator.summary;
  }
}

// Stream summary as tokens (frontend expectation)
if (finalSummary) {
  const words = finalSummary.split(' ');
  for (let i = 0; i < words.length; i += 5) {
    const tokenChunk = words.slice(i, i + 5).join(' ') + ' ';
    res.write(`data: ${JSON.stringify({
      type: 'token',
      data: { token: tokenChunk },
      timestamp: Date.now()
    })}\n\n`);
    
    // Simulate natural typing (10ms delay per chunk)
    await new Promise(resolve => setTimeout(resolve, 10));
  }
}

// Send completion with summary and services
res.write(`data: ${JSON.stringify({
  type: 'message_complete',
  data: {
    conversationId: convId,
    summary: finalSummary,
    services: finalServices
  },
  timestamp: Date.now()
})}\n\n`);
```

**Architectural Benefits:**
- Progressive rendering improves perceived performance
- Token streaming matches LLM UX patterns users expect
- Fallback to full summary if streaming fails

### Fix #3: Error Deduplication

**File:** [GeoAIStreamingHandler.ts](file://e:/codes/GeoAI-UP/server/src/llm-interaction/handlers/GeoAIStreamingHandler.ts#L75-L95)

**Strategy:** Time-based deduplication to filter duplicate error callbacks from LangGraph.

```typescript
export class GeoAIStreamingHandler extends BaseCallbackHandler {
  private lastErrorTimestamp: number = 0;
  private errorDeduplicationWindow: number = 100; // ms

  async handleChainError(error: Error): Promise<void> {
    const now = Date.now();
    
    // Skip if same error occurred within deduplication window
    if (now - this.lastErrorTimestamp < this.errorDeduplicationWindow) {
      return; // Suppress duplicate
    }
    
    this.lastErrorTimestamp = now;
    
    this.writeSSE({
      type: 'error',
      message: error.message,
      stack: process.env.NODE_ENV === 'development' ? error.stack : undefined,
      timestamp: now,
    });
  }
}
```

**Why This Works:**
- LangGraph duplicate callbacks occur within same millisecond
- 100ms window is safe (real errors won't occur that fast)
- Preserves legitimate rapid sequential errors (unlikely but possible)

## 🎯 Architectural Principles Applied

### 1. **Separation of Concerns**
- Backend handles workflow orchestration
- Frontend handles user experience
- Clear contract via SSE event types

### 2. **Progressive Enhancement**
- Token streaming provides better UX than instant display
- Workflow events available for debugging without breaking UI
- Graceful fallbacks at every level

### 3. **Defensive Programming**
- Error deduplication prevents UI spam
- Summary fallback ensures content always displays
- Type checking prevents runtime errors

### 4. **Single Source of Truth**
- Summary generated once in `SummaryGenerator` node
- Captured and streamed by controller
- Displayed by frontend from single source

## 📝 Lessons Learned

### Key Insight #1: Event Contract Design
**Lesson:** Backend and frontend must agree on event vocabulary from the start.

**Anti-pattern observed:**
- Backend evolved independently with internal event types
- Frontend assumed standard chat event types
- No integration testing until production issue

**Best practice:**
- Define SSE event schema in shared TypeScript types
- Validate both sides against schema during development
- Document event flow in architecture docs

### Key Insight #2: Debugging vs. Production Events
**Lesson:** Separate debug events from user-facing events.

**Solution implemented:**
- Workflow events (`step_start`, `step_complete`) kept for debugging
- User events (`message_start`, `token`, `message_complete`) for UI
- Frontend safely ignores unknown event types

### Key Insight #3: Content Must Flow End-to-End
**Lesson:** Generating content ≠ delivering content.

**Failure mode:**
- Summary generated in LangGraph node ✅
- Stored in workflow state ✅
- **Never extracted and sent to client** ❌

**Prevention:**
- Trace data flow from generation to delivery
- Add assertions/checks at each boundary
- Test with real data, not just mocks
