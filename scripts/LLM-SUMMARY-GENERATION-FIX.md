# Chat Workflow - LLM-Generated Summaries & Streaming Fixes

## Date
May 5, 2026

## Executive Summary

Implemented two critical improvements to the chat workflow:
1. **Stopped leaking internal LLM tokens** - Disabled intermediate token streaming from GeoAIStreamingHandler
2. **LLM-generated summaries** - Replaced template substitution with intelligent, natural language summaries generated by LLM

These changes transform the chat experience from robotic template output to conversational, context-aware responses.

---

## 🔍 Problem #1: Internal Data Leaking to UI

### Symptom

Browser console flooded with JSON metadata instead of natural language:
```javascript
[Chat Store] Token event missing data field 
{type: 'token', content: ',\n        "includeMaps": true,\n        "'}
{type: 'token', content: 'author": "GeoAI-UP System",...'}
{type: 'token', content: 'pluginId": "report_generator",...'}
```

### Root Cause

**GeoAIStreamingHandler.handleLLMNewToken()** was emitting tokens for **every LLM invocation** in the workflow:
- Goal Splitter's LLM call
- Task Planner's LLM call  
- Any other agent using LLM

These internal calls were streaming their raw outputs (often JSON structures, metadata, etc.) directly to the frontend.

**Architecture Issue:**
```
LangGraph Workflow
├── Goal Splitter → LLM call → tokens streamed ❌ (internal)
├── Task Planner → LLM call → tokens streamed ❌ (internal)
├── Plugin Executor → No LLM
└── Summary Generator → LLM call → tokens streamed ✅ (should be only this)
```

All LLM tokens were being sent to the client, not just the final summary.

### Solution: Disable Intermediate Token Streaming

**File:** [GeoAIStreamingHandler.ts](file://e:/codes/GeoAI-UP/server/src/llm-interaction/handlers/GeoAIStreamingHandler.ts#L19-L29)

```typescript
/**
 * Called when LLM generates a new token
 * DISABLED: We manually stream only the final summary to avoid leaking internal LLM calls
 */
async handleLLMNewToken(token: string): Promise<void> {
  // Skip intermediate LLM tokens - only stream final summary from ChatController
  // this.writeSSE({
  //   type: 'token',
  //   content: token,
  //   timestamp: Date.now(),
  // });
}
```

**Result:** Only the curated summary from ChatController is streamed to the frontend.

## 🔍 Problem #2: Robotic Template-Based Summaries

### Symptom

UI displayed templated text with variable substitution:
```markdown
Generate a friendly summary of the analysis results.

Goals completed: 1
Goals failed: 1

Results: Total: 2, Success: 1, Failed: 1...

Provide a concise, helpful summary...
```

This is the **raw template** with variables replaced - not a natural language summary.

### Root Cause

**SummaryGenerator** was using simple string replacement:
```typescript
// Old approach - template substitution
summary = summary.replace(/\{completedGoals\}/g, '1')
summary = summary.replace(/\{resultsSummary\}/g, 'Total: 2, Success: 1...')
```

**Problems:**
- ❌ Robotic, formulaic language
- ❌ No contextual understanding
- ❌ Can't explain failures intelligently
- ❌ Doesn't adapt to different scenarios
- ❌ Exposes internal structure to users

### Desired Behavior

Natural, conversational summary like:
```markdown
I've completed the buffer analysis on your data. Here's what happened:

✅ **Successfully processed** the Wuhu Linhe river dataset, creating a 1km buffer zone around 263 features. The analysis generated polygon geometries covering the area you requested.

⚠️ **One issue encountered**: The MVT tile publishing step failed because the intermediate buffer result couldn't be directly converted to vector tiles. This is a known limitation with temporary processing results.

📊 **What you can access**: The buffer analysis results are available as a GeoJSON file with full attribute data including river names, types, and geometric properties.

Would you like me to help you visualize these results or perform additional analysis on the buffered areas?
```

#### Change #2: Try LLM First, Fallback to Template

```typescript
async generate(state: GeoAIStateType, options: SummaryOptions = {}): Promise<string> {
  try {
    // Try LLM-based generation first (if API keys configured)
    if (this.llmConfig) {
      console.log('[Summary Generator] Using LLM for natural language summary');
      return await this.generateWithLLM(state, language);
    }
    
    // Fallback to template-based generation
    console.log('[Summary Generator] LLM not configured, using template fallback');
    const template = await this.loadSummaryTemplate(language);
    
    if (template) {
      return this.generateFromTemplate(state, template, options);
    } else {
      return this.generateFallback(state, options);
    }
  } catch (error) {
    console.error('[Summary Generator] Error generating summary:', error);
    return this.generateFallback(state, options);
  }
}
```

#### Change #3: Implement LLM Generation

```typescript
private async generateWithLLM(state: GeoAIStateType, language: string): Promise<string> {
  // Prepare rich context for LLM
  const context = this.prepareLLMContext(state);
  
  // Load prompt template (now used as instruction, not fill-in-the-blanks)
  const promptTemplate = await this.promptManager.loadTemplate('response-summary', language);
  
  // Create LLM instance
  const llm = LLMAdapterFactory.createAdapter(this.llmConfig!);
  
  // Create chain: prompt + LLM
  const chain = promptTemplate.pipe(llm);
  
  // Invoke LLM with context
  const response = await chain.invoke(context);
  
  // Extract text from various response formats
  let summary: string;
  if (typeof response === 'string') {
    summary = response;
  } else if (Array.isArray(response)) {
    // Handle array of content blocks
    summary = response
      .map(block => {
        if (typeof block === 'string') return block;
        if (block && typeof block === 'object' && 'text' in block) return block.text;
        return '';
      })
      .join('');
  } else if (response && typeof response === 'object' && 'content' in response) {
    summary = String(response.content);
  } else {
    summary = String(response);
  }
  
  console.log('[Summary Generator] LLM generated summary successfully');
  return summary;
}
```

#### Change #4: Prepare Rich Context for LLM

```typescript
private prepareLLMContext(state: GeoAIStateType): Record<string, any> {
  const context: Record<string, any> = {};
  
  // Goals information
  if (state.goals && state.goals.length > 0) {
    context.completedGoals = state.goals.length.toString();
    context.goalsList = state.goals.map(g => `- ${g.description} (${g.type})`).join('\n');
  }
  
  // Results with success/failure details
  if (state.executionResults) {
    const results = Array.from(state.executionResults.values());
    const successCount = results.filter(r => r.status === 'success').length;
    const failCount = results.filter(r => r.status === 'failed').length;
    
    context.successCount = successCount.toString();
    context.failCount = failCount.toString();
    
    // Detailed successful operations
    const successOps = results.filter(r => r.status === 'success');
    context.successfulOperations = successOps.length > 0
      ? successOps.map(r => `- ${r.metadata?.pluginId}: Completed successfully`).join('\n')
      : 'None';
    
    // Detailed failed operations with error messages
    const failedOps = results.filter(r => r.status === 'failed');
    context.failedOperations = failedOps.length > 0
      ? failedOps.map(r => `- ${r.metadata?.pluginId}: ${r.error}`).join('\n')
      : 'None';
  }
  
  // Services available to user
  if (state.visualizationServices && state.visualizationServices.length > 0) {
    context.serviceCount = state.visualizationServices.length.toString();
    context.servicesList = state.visualizationServices.map(s => 
      `- ${s.type.toUpperCase()} Service: ${s.url}`
    ).join('\n');
  }
  
  // Errors and warnings
  if (state.errors && state.errors.length > 0) {
    context.errorsList = state.errors.map(e => `- ${e.goalId}: ${e.error}`).join('\n');
  }
  
  return context;
}
```

## 🏗️ Architectural Benefits

### 1. **Separation of Concerns**

| Component | Responsibility |
|-----------|---------------|
| **PromptManager** | Loads and converts templates |
| **SummaryGenerator** | Orchestrates summary generation |
| **LLM** | Generates natural language |
| **ChatController** | Streams final summary to client |

Each component has a single, clear responsibility.

### 2. **Graceful Degradation**

```
LLM Available? 
├── Yes → Generate natural language summary ✅
└── No → Fallback to template substitution ⚠️
          └── Template missing? → Use hardcoded fallback ⚠️
```

System works at all capability levels.

### 3. **Rich Context for LLM**

Instead of just passing counts, we provide:
- ✅ Goal descriptions and types
- ✅ Detailed success/failure information
- ✅ Specific error messages
- ✅ Service URLs and metadata
- ✅ Complete execution history

This enables the LLM to generate **contextually aware** summaries.

### 4. **Flexible Response Handling**

The code handles multiple LLM response formats:
- Plain strings
- Arrays of content blocks
- Objects with content property
- Other formats via String() conversion

This ensures compatibility with different LLM providers.

## 🎯 Key Insights

### Insight #1: Don't Stream Internal State

**Lesson:** Just because you CAN stream something doesn't mean you SHOULD.

**Anti-pattern:**
```typescript
// Streaming every LLM token from every workflow step
handleLLMNewToken(token) {
  streamToClient(token);  // ❌ Leaks internals
}
```

**Best Practice:**
```typescript
// Curate what gets streamed
handleLLMNewToken(token) {
  // Skip - only stream final summary
}

// Later, explicitly stream summary
streamToClient(finalSummary);  // ✅ Controlled output
```

---

### Insight #2: Templates vs. LLM Generation

**Template Substitution:**
- ✅ Fast, deterministic
- ✅ No API costs
- ❌ Robotic, inflexible
- ❌ Exposes internal structure

**LLM Generation:**
- ✅ Natural, conversational
- ✅ Contextually aware
- ✅ Adapts to different scenarios
- ❌ Requires API keys
- ❌ Slower, non-deterministic

**Solution:** Use both - LLM when available, template as fallback.

---

### Insight #3: Rich Context Enables Better Responses

**Poor Context:**
```
Goals: 1
Success: 1
Fail: 1
```

**Rich Context:**
```
Goals:
- Buffer analysis on Wuhu Linhe river (spatial_analysis)

Success:
- Buffer Analysis plugin: Created 1km buffer around 263 features
  Result: GeoJSON with polygons, EPSG:4326, bbox: [129.66, 44.74, 130.20, 45.08]

Fail:
- MVT Publisher: "Unsupported data source type: step_buffer_1000m_wuhulinhe"
  Reason: Temporary processing results can't be directly published as MVT
```

The LLM can generate much better summaries with rich context.

Cache generated summaries to avoid redundant LLM calls:
```typescript
private summaryCache = new Map<string, string>();

async generate(state: GeoAIStateType): Promise<string> {
  const cacheKey = this.generateCacheKey(state);
  
  if (this.summaryCache.has(cacheKey)) {
    return this.summaryCache.get(cacheKey)!;
```
