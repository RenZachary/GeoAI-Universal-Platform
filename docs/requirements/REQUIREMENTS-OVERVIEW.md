GeoAI-UP（GeoAI Universal Platform）平台详细需求规格说明书
一、平台概述
GeoAI-UP 全称 GeoAI Universal Platform，是一款面向实际应用场景的地理信息AI综合应用平台。平台核心能力为：支持用户通过自然语言输入，对主流地理信息数据源进行解析，由大语言模型（LLM）自动拆解用户需求、调用对应插件，完成地理信息分析、数据渲染等基础功能，同时支持非地理信息类对话交互，兼顾实用性与易用性。
平台核心目标：降低地理信息分析门槛，实现“自然语言交互+自动化GIS分析”，支持自定义扩展，可快速打包部署，满足各类地理信息相关应用场景需求。
注：平台不涉及用户管理、权限管理相关功能，聚焦核心业务能力实现。
二、核心能力需求
2.1 自然语言交互能力
- 支持中英文自然语言输入，可识别地理信息相关需求（如空间分析、数据渲染、报告生成）及非地理信息相关对话（如基础咨询、指令确认）。
- 自然语言对话必须采用流式输出，确保交互响应流畅，提升用户体验。
- 支持多轮对话，可上下文关联，记忆当前对话场景中的用户需求、数据源选择、分析参数等信息。
- 支持历史对话管理，可查看、回溯过往对话记录及对应分析结果、报告等内容。
- 友好的应答机制：针对系统错误、LLM配置异常、数据源异常、插件调用失败等场景，需给出清晰、易懂的中英文应答提示，引导用户排查问题。
2.2 大语言模型（LLM）能力
- 支持多类大语言模型，至少包含 qwen 系列模型，预留扩展接口可新增其他主流LLM。
- 前端支持LLM切换、参数配置（如模型地址、API密钥、请求超时时间等），配置操作简洁易懂，无需后台修改代码。
- 支持独立的LLM提示词模板管理，可新增、编辑、删除提示词模板，模板需适配地理信息分析场景，提升LLM对GIS需求的解析准确率。
- 由LLM负责拆解用户自然语言需求，核心是先识别需求中的多个输出目标并拆分，再对每个输出目标单独拆解为可执行的子目标、填充相关参数；例如用户输入“请显示小区数据，显示河流的500米缓冲区数据”，LLM需先识别出两个独立输出目标，第一个目标（显示小区数据）的执行流程由LLM进一步拆解（如“加载小区数据源→解析数据→发布MVT服务→前端展示”）并填充对应参数，第二个目标（显示河流500米缓冲区数据）按相同逻辑单独拆解、填充参数，确保每个输出目标都能独立执行并达成。
- 单个自然语言对话请求，允许LLM多次调用（如需求拆解后，多次调用不同插件执行子目标，或多次调整分析参数）。
- 充分利用 langchain 框架，实现LLM与插件、数据源的联动，优化需求拆解效率和执行流程。
2.3 地理信息数据处理能力
2.3.1 数据源支持
- 支持主流地理信息数据源格式，包括但不限于：shapefile（多文件）、GeoJSON、PostGIS（数据库）、tif（影像），预留扩展接口支持其他常见GIS格式。
- 支持用户前端上传本地数据源，其中shapefile格式需支持多文件（.shp、.shx、.dbf等）批量上传，确保文件完整性校验。
- 支持配置0个或者多个PostGIS数据源，前端可管理（新增、编辑、删除）PostGIS连接信息（地址、端口、数据库名、账号密码等），实现多数据源切换使用。
2.3.2 数据处理规则
- 系统运行期间，必须保持原始数据格式（NativeData），严禁随意转换：PostGIS数据源的操作直接在数据库内部执行查询、分析，必要时可通过临时表完成中间转换操作，不改变原始数据；其他格式数据源（如shapefile、GeoJSON）保持原始文件结构，仅在分析时生成临时文件。
- 严格规范数据类型定义，在开发阶段（编译时）暴露数据类型不匹配、数据格式错误等问题，避免运行时因数据类型问题导致系统异常。
2.3.3 空间分析与可视化
- 支持常见地理空间分析功能，所有空间分析、报告生成、可视化方法均封装为内置（buildin）插件，可直接被LLM调用。
- 支持自定义插件开发与接入，插件可实现空间分析功能、数据处理、格式转换等扩展能力，满足个性化需求。
- 支持多目标分析：单个自然语言请求可触发多个分析任务，生成多个分析结果（如同时生成空间分布图、统计表格、分析报告），支持多结果并行展示。
- 空间数据与影像数据展示规范：
       
  - 矢量空间数据（shapefile、GeoJSON、PostGIS导出矢量等）统一发布为MVT服务，用于前端浏览；
  - 影像数据（tif等）统一发布为WMS服务，用于前端浏览；
  - 热力图可视化可直接使用GeoJSON格式，无需转换为MVT服务。
2.3.4 报告生成
- 支持基于分析结果自动生成报告，报告内容需包含分析目标、数据源信息、分析过程、结果统计、可视化图表（关联MVT/WMS服务或热力图）等核心内容。
- 报告支持导出，导出格式需适配常见文档格式（如PDF、HTML），确保报告内容完整、排版规范。
2.4 工作目录管理
平台需实现统一的工作目录管理，目录结构固定且清晰，所有相关文件（数据、配置、插件等）均纳入对应目录，确保系统可维护性，具体目录用途如下：
- 本地数据目录：存储用户上传的本地数据源（shapefile、GeoJSON、tif等），按数据源类型、上传时间分类管理；
- LLM配置目录：存储LLM的配置文件、提示词模板等相关内容；
- 插件目录：存储用户上传的自定义插件、系统内置插件，按插件类型分类管理；
- 系统数据库目录：存储系统核心配置、历史对话记录、分析任务记录等，使用sqlite数据库；
- 临时目录：存储系统运行过程中生成的临时文件（如临时入库数据、插件运行临时结果等），系统退出或任务完成后可自动清理（可配置清理策略）；
- 分析结果持久化目录：存储分析完成后的最终结果，包括GeoJSON、shapefile格式结果数据、MVT服务元信息、WMS服务元信息、生成的报告文件等，支持长期查询和复用。
三、前端需求
3.1 基础交互
- 支持中英文切换，包括前端UI界面（菜单、按钮、提示信息等）和后端自然语言应答内容，切换后实时生效，无需重启系统。
- 界面设计友好、简洁，操作流程清晰。
3.2 数据管理
- 支持数据源管理：前端可上传本地数据源（shapefile多文件、GeoJSON、tif等），显示数据源名称、格式、大小、上传时间等信息，支持数据源删除、预览。
- 支持PostGIS数据源配置：可新增、编辑、删除多个PostGIS数据源连接信息，支持连接测试，确保数据源可正常访问。
3.3 插件管理
- 支持插件上传：前端可上传自定义插件（需符合平台插件规范），上传后自动校验插件合法性。
- 支持插件生命周期管理：可对已上传的插件进行启动、停用、删除操作，实时显示插件运行状态（启用/停用）。
- 支持插件列表展示：显示插件名称、类型、版本、描述、运行状态等信息，便于用户快速管理。
3.4 空间数据浏览
- 支持MVT服务（矢量数据）、WMS服务（影像数据）的加载与浏览，可实现地图缩放、平移、图层控制（显示/隐藏）等基础操作。
- 支持热力图（GeoJSON格式）展示，可调整热力图参数（如半径、透明度），提升可视化效果。
- 支持分析结果可视化展示，可关联对应的MVT/WMS服务或热力图，直观呈现分析结果。
3.5 LLM配置
- 支持LLM模型切换：前端可选择已支持的LLM模型（如qwen），切换后立即生效。
- 支持LLM参数配置：可配置模型地址、API密钥、请求超时时间、上下文长度等参数，配置后可保存，下次启动无需重新配置。
3.6 对话与结果管理
- 提供自然语言对话输入框，支持流式输出显示，可实时查看对话响应内容。
- 支持历史对话管理：可查看所有历史对话记录，点击某条记录可回溯对话内容及对应分析结果、报告。
- 支持分析结果、报告的查看与导出，导出操作简单便捷。
四、后端需求
4.1 技术架构
- 遵循文档驱动开发（DDD）模式（非领域驱动开发），开发过程中需完善相关文档，确保代码与文档一致。
- 后台逻辑层次清晰划分，至少包含：接口层、LLM交互层、插件调度层、数据接入层、空间分析层、存储层，各层职责明确，低耦合、高内聚。
- 每一层必须通过工厂类实现，例如数据接入层需实现 DataAccessorFactory，插件调用GeoJSON、PostGIS相关方法时，必须通过该工厂类获取对应的数据访问实例，统一数据访问入口.
4.2 核心模块
4.2.1 LLM交互模块
- 集成 langchain 框架，实现LLM的调用、提示词模板管理、需求拆解等功能。
- 支持多LLM模型适配，提供统一的LLM调用接口，新增LLM模型时无需大幅修改核心代码。
- 实现LLM流式输出功能，与前端交互，确保对话响应流畅。
4.2.2 数据接入模块
- 实现多种数据源（shapefile、GeoJSON、PostGIS、tif）的接入与解析，确保数据读取准确。
- 通过 DataAccessorFactory 统一管理数据访问实例，实现不同数据源的统一访问接口。
- 实现跨数据源操作逻辑，遵循“性能优先”原则，完成多数据源联合分析的中间处理（如GeoJSON临时入库）。
- 实现数据类型校验，在编译时暴露数据类型相关问题，确保数据处理的安全性和准确性。
4.2.3 插件调度模块
- 实现插件的加载、启动、停用、删除等生命周期管理，提供插件调用接口，支持LLM动态调用插件。
- 实现内置插件（空间分析、报告生成、可视化）的集成，确保插件可正常被调用且结果准确。
- 支持自定义插件接入，提供插件开发规范，确保自定义插件与系统兼容。
- 插件数据源传入和传出，都通过NativeData，（可视化插件除外，可视化插件可以是MVT、WMS、Geojson（只有热力图可以是Geojson））
4.2.4 空间分析与可视化模块
- 基于内置插件，实现常见空间分析功能（如叠加分析、缓冲区分析、统计分析等），确保分析结果准确。
- 实现MVT服务、WMS服务的发布功能，确保前端可正常加载、浏览空间数据和影像数据。
- 实现热力图数据处理，支持GeoJSON格式热力图生成，适配前端可视化需求。
4.2.5 存储模块
- 管理平台统一工作目录，实现各目录下文件的创建、读取、更新、删除操作，确保文件存储规范。
- 使用sqlite数据库存储系统配置、历史对话、任务记录等核心数据，确保数据持久化。
- 实现临时文件管理，支持临时文件自动清理，避免占用过多存储空间。
4.2.6 接口模块
- 提供RESTful API接口，支持前端与后端的数据交互、指令调用（如数据源上传、LLM配置、插件管理、分析任务提交等）。
- 接口支持中英文应答，针对异常请求（如参数错误、数据源异常）返回清晰的错误提示。
- 确保接口性能稳定，支持并发请求，大数据量分析时不出现接口超时、崩溃等问题。
五、部署与构建需求
- 支持一键构建为包含nodejs运行时的部署包，无需用户额外安装nodejs环境，可直接运行。
- 打包脚本参考路径：E:\GeoAI-Universal-Platform\package.js，确保打包流程顺畅，部署包包含所有必要的依赖、配置文件、内置插件等。
- 部署包支持跨平台运行（Windows、Linux），打包时需考虑不同系统的兼容性。
- 提供部署说明文档，明确部署步骤、启动命令、常见问题排查方法。
六、开发规范需求
- 严格遵循文档驱动开发（DDD）模式，开发前完善需求文档、设计文档，开发过程中同步更新文档，确保文档与代码一致。
- 后端使用TypeScript开发，严格规范代码风格，遵循ESLint、Prettier规范，避免代码冗余、格式混乱。
- 插件开发需遵循平台统一规范，确保插件可正常接入、调用，且具有良好的可维护性。
- 所有核心模块、工具类需编写单元测试，确保代码质量，降低运行时异常概率。
- 严格遵循工厂类设计模式，每一层均通过工厂类实现实例创建，统一接口，降低模块间耦合。
七、目录结构需求
平台目录结构固定，如下所示，各目录职责明确，不得随意修改目录名称及层级：
- docs/：存储平台相关文档，包括需求规格说明书、设计文档、部署文档、插件开发规范等；
- server/：存储后端代码，包括各核心模块（LLM交互、数据接入、插件调度等）、配置文件、系统数据库等；
- web/：存储前端代码，包括界面组件、交互逻辑、路由配置、国际化资源等；
- build/：存储打包脚本，包括一键构建部署包的相关脚本、配置文件；
- vendor/：存储第三方依赖包，包括后端、前端所需的第三方库、工具等；
- scripts/：存储零散脚本，包括临时数据处理脚本、测试脚本等；
- README.md：平台说明文档，包括平台简介、快速启动、核心功能、注意事项等内容。
