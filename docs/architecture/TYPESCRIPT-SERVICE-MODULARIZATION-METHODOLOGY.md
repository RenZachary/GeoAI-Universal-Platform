# TypeScript 服务层模块化重构方法论

## 1. 重构目标

将单体服务文件拆分为职责清晰的模块结构，提升代码可维护性、可测试性和可扩展性。

## 2. 核心原则

### 2.1 单一职责原则（SRP）
- 每个模块只负责一个明确的功能领域
- 避免模块承担多重职责导致耦合度增加
- 功能边界清晰，便于定位和修改

### 2.2 依赖倒置原则（DIP）
- 高层模块不依赖低层模块的具体实现
- 通过接口或抽象类解耦模块间依赖
- 依赖通过构造函数注入，便于测试和替换

### 2.3 最小化导出原则
- `index.ts` 仅导出外部必需的公共 API
- 内部实现细节对外部不可见
- 类型定义按需导出，避免暴露内部数据结构

### 2.4 关注点分离（SoC）
- 业务逻辑、数据访问、验证逻辑分离到不同模块
- 错误处理独立管理，统一错误类型体系
- 常量配置集中管理，避免硬编码

## 3. 模块划分策略

### 3.1 主服务模块（Service.ts）
**职责**：
- 协调各子模块完成业务流程
- 提供统一的对外接口
- 管理模块生命周期和状态

**不包含**：
- 具体业务逻辑实现
- 错误类定义
- 工具函数

### 3.2 错误处理模块（errors.ts）
**职责**：
- 定义自定义错误类层次结构
- 提供标准化的错误码和消息
- 支持错误分类和捕获

**原则**：
- 错误类继承自标准 Error
- 包含明确的错误码标识
- 便于上层统一处理

### 3.3 接口定义模块（interface.d.ts）
**职责**：
- 定义数据传输对象（DTO）
- 定义类型别名和接口
- 提供类型安全保障

**原则**：
- 仅包含类型定义，无实现代码
- 使用 `.d.ts` 或 `.ts` 文件
- 内部类型不对外导出

### 3.4 常量配置模块（constant.ts）
**职责**：
- 集中管理魔法数字和字符串
- 定义阈值、超时时间等配置
- 提供默认值和边界条件

**原则**：
- 使用 `const` 声明不可变常量
- 命名语义化，避免缩写
- 按功能分组组织

### 3.5 功能子模块
**职责**：
- 封装特定领域的业务逻辑
- 实现独立的可复用功能
- 通过依赖注入获取所需资源

**常见子模块**：
- **Validator**：输入验证和业务规则校验
- **Connector**：外部系统连接管理
- **Discoverer**：数据发现和元数据提取
- **Registrar**：数据注册和持久化
- **Calculator**：复杂计算和数据处理

## 4. 依赖管理规范

### 4.1 模块间依赖方向
```
Service (协调层)
  ↓
Validator / Connector / Discoverer / Registrar / Calculator (功能层)
  ↓
errors / interface / constant (基础层)
```

### 4.2 禁止的依赖关系
- 功能模块之间不应相互依赖
- 基础模块不应依赖功能模块
- 避免循环依赖

### 4.3 依赖注入方式
- 通过构造函数传递依赖
- 避免在模块内部直接实例化其他模块
- 便于单元测试时替换 mock 对象

## 5. 访问控制策略

### 5.1 公共 API 设计
- 仅暴露必要的类和方法
- 使用 `public`/`private`/`protected` 明确访问级别
- 内部方法标记为 `private`

### 5.2 导出控制
- `index.ts` 作为唯一导出入口
- 不导出内部实现类和工具函数
- 类型定义按需导出

### 5.3 ESLint 规则配合
- 配置 `no-restricted-imports` 限制直接导入内部模块
- 强制通过 `index.ts` 统一访问
- 防止架构混乱和循环依赖

## 6. 错误处理规范

### 6.1 错误类层次
```
Error (基类)
  └─ DataSourceError (领域错误)
      ├─ ConnectionError (连接错误)
      └─ ValidationError (验证错误)
```

### 6.2 错误抛出原则
- 使用具体的错误类而非通用 Error
- 提供有意义的错误消息
- 包含错误码便于上层识别

### 6.3 错误传播
- 底层模块抛出具体错误
- 中间层可选择捕获并包装
- 顶层统一处理和响应

## 7. 重构步骤

### 7.1 分析阶段
1. 识别单体文件中的职责边界
2. 提取独立的业务逻辑块
3. 确定模块间依赖关系

### 7.2 设计阶段
1. 规划模块结构和文件组织
2. 定义模块间接口和契约
3. 设计错误处理体系

### 7.3 实施阶段
1. 创建基础模块（errors、interface、constant）
2. 拆分功能子模块
3. 重构主服务类为协调层
4. 更新导入路径和依赖注入

### 7.4 验证阶段
1. 确保所有引用正确更新
2. 运行类型检查和 lint 规则
3. 执行单元测试验证功能
4. 检查循环依赖和架构违规

## 8. 质量保证

### 8.1 代码审查要点
- 模块职责是否清晰
- 依赖关系是否合理
- 导出是否符合最小化原则
- 错误处理是否完善

### 8.2 测试策略
- 每个功能模块独立单元测试
- 主服务类集成测试
- 错误场景覆盖测试

### 8.3 文档要求
- 模块职责说明
- 公共 API 文档
- 依赖关系图
- 使用示例

## 9. 适用场景

### 9.1 适合重构的情况
- 服务类超过 500 行代码
- 包含多个独立的功能领域
- 需要提高可测试性
- 团队协作需要清晰边界

### 9.2 不适合重构的情况
- 简单的 CRUD 服务
- 代码量少于 200 行
- 短期内不会扩展功能
- 项目处于快速原型阶段

## 10. 注意事项

### 10.1 避免过度设计
- 不要为简单逻辑创建过多模块
- 保持模块粒度适中
- 权衡复杂度和可维护性

### 10.2 保持向后兼容
- 公共 API 尽量保持不变
- 提供迁移指南
- 逐步废弃旧接口

### 10.3 性能考虑
- 模块拆分不影响运行时性能
- 避免不必要的对象创建
- 注意懒加载和缓存策略
