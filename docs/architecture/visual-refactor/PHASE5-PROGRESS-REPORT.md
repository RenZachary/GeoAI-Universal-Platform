#### Validation Rules Implemented

**Rule 1: At Most One Terminal Node**
```typescript
// A goal can have AT MOST ONE terminal node
if (terminalNodeCount > 1) {
  console.error(`Plan has ${terminalNodeCount} terminal nodes (max 1 allowed)`);
  return null; // Reject plan
}
```

**Rationale:** Terminal nodes (visualization, textual) produce final output. Multiple terminals would create conflicting outputs.

**Example Violation:**
```json
{
  "steps": [
    { "pluginId": "choropleth_renderer" },  // ← Terminal #1
    { "pluginId": "report_generator" }       // ← Terminal #2 ❌
  ]
}
```

---

**Rule 2: Terminal Node Must Be Last Step**
```typescript
// Terminal nodes MUST be the last step
if (terminalNodeCount === 1 && terminalNodeIndex !== plan.steps.length - 1) {
  // Auto-fix: Move terminal node to end
  const fixedSteps = [...plan.steps];
  const terminalStep = fixedSteps.splice(terminalNodeIndex, 1)[0];
  fixedSteps.push(terminalStep);
  return { ...plan, steps: fixedSteps };
}
```

**Rationale:** Terminal nodes consume all previous results. Nothing can execute after them.

**Auto-Fix Example:**
```json
// Before (invalid):
{
  "steps": [
    { "pluginId": "buffer_analysis" },
    { "pluginId": "choropleth_renderer" },  // ← Terminal in middle ❌
    { "pluginId": "statistics_calculator" }
  ]
}

// After (auto-fixed):
{
  "steps": [
    { "pluginId": "buffer_analysis" },
    { "pluginId": "statistics_calculator" },
    { "pluginId": "choropleth_renderer" }   // ← Moved to end ✅
  ]
}
```

---

**Rule 3: Textual Plugins Require Predecessor**
```typescript
// Textual plugins need non-terminal input
const textualPluginIds = ['report_generator'];
if (textualPluginIds.includes(terminalStep.pluginId) && plan.steps.length === 1) {
  console.error(`Textual plugin requires a predecessor step`);
  return null; // Reject plan
}
```

**Rationale:** Textual plugins (like report generators) need execution results as input. They cannot run standalone.

**Example Violation:**
```json
{
  "steps": [
    { "pluginId": "report_generator" }  // ❌ No input data to generate report from
  ]
}
```

**Valid Plan:**
```json
{
  "steps": [
    { "pluginId": "statistics_calculator" },  // ← Produces statistics
    { "pluginId": "report_generator" }         // ← Uses statistics to generate report ✅
  ]
}
```

#### Method: `validateTerminalNodeConstraints()`

**Signature:**
```typescript
private validateTerminalNodeConstraints(
  plan: ExecutionPlan,
  compatiblePluginIds: string[]
): ExecutionPlan | null
```

**Parameters:**
- `plan`: The execution plan generated by LLM (Stage 2)
- `compatiblePluginIds`: List of plugin IDs that passed Stage 1 filtering

**Returns:**
- `ExecutionPlan`: Validated (and possibly auto-fixed) plan
- `null`: Plan is invalid and cannot be fixed

**Process Flow:**
```
1. Check if plan is empty → Return as-is
2. Identify terminal plugin IDs from capability registry
3. Count terminal nodes in plan
4. Apply Rule 1: Reject if >1 terminal nodes
5. Apply Rule 2: Auto-fix if terminal not last
6. Apply Rule 3: Reject if textual plugin has no predecessor
7. Return validated plan
```

#### Method: `getTerminalPluginIds()`

**Purpose:** Dynamically identify which plugins are terminal nodes based on their capabilities.

**Implementation:**
```typescript
private getTerminalPluginIds(pluginIds: string[]): string[] {
  const terminalCategories = ['visualization', 'textual'];
  
  const terminalIds: string[] = [];
  
  for (const pluginId of pluginIds) {
    const capability = PluginCapabilityRegistry.getCapability(pluginId);
    if (capability && terminalCategories.includes(capability.executionCategory)) {
      terminalIds.push(pluginId);
    }
  }
  
  return terminalIds;
}
```

**Advantages:**
- ✅ Dynamic: Automatically adapts when new plugins are registered
- ✅ Centralized: Uses PluginCapabilityRegistry as single source of truth
- ✅ Maintainable: No hard-coded plugin ID lists

## Architecture Improvements

### Three-Stage Decision Process

**Before Phase 5:**
```
Stage 1: Filter → Stage 2: Select → Execute
                              ↑
                        No validation!
```

**After Phase 5:**
```
Stage 1: Filter → Stage 2: Select → Stage 3: Validate → Execute
                                         ↓
                                   - Terminal node check
                                   - Auto-fix ordering
                                   - Predecessor validation
```

### Benefits

1. **Correctness Guarantee**: Invalid plans are rejected before execution
2. **Auto-Repair**: Some violations (wrong ordering) are automatically fixed
3. **Clear Error Messages**: Detailed logging explains why plans fail
4. **Extensible**: Easy to add more validation rules in Stage 3

## Architect's Notes

### Strengths of Three-Stage Design

1. **Progressive Refinement**: Each stage narrows down options
   - Stage 1: 20+ plugins → 3-5 candidates
   - Stage 2: 3-5 candidates → 1 selected
   - Stage 3: 1 selected → Validated or rejected

2. **Separation of Concerns**:
   - Stage 1: Deterministic rules (fast, reliable)
   - Stage 2: Semantic understanding (flexible, intelligent)
   - Stage 3: Constraint enforcement (safe, correct)

3. **Graceful Degradation**: If any stage fails, fallback to safe defaults

### Potential Issues

**Issue 1: Over-Strict Validation**
- Risk: Valid but unusual plans might be rejected
- Mitigation: Log warnings instead of errors for borderline cases
- Future: Add configuration to relax certain rules

**Issue 2: Auto-Fix May Hide LLM Errors**
- Risk: LLM learns it can produce invalid plans (system will fix them)
- Mitigation: Log auto-fixes prominently for monitoring
- Future: Feed validation errors back to LLM for learning

**Issue 3: Hard-Coded Textual Plugin List**
- Current: `['report_generator']`
- Risk: New textual plugins won't be caught
- Mitigation: Use PluginCapabilityRegistry to identify textual category
- TODO: Update `getTerminalPluginIds()` to include textual category check
